# PubChem Agent Meeting Notes — Overview and MVP

[← Оглавление](../../pubchem-agent-meeting-notes.md) | [Следующий раздел →](./02-tools-and-agent-loop.md)

## Проект

**Название:** PubChem ИИ-агент (Request to PubChem)  
**Категория:** information retrieval  
**Источник данных:** PubChem / PUG REST API  
**Цель:** пользователь описывает молекулу или ее свойства текстом, агент интерпретирует запрос, вызывает нужные инструменты и возвращает найденные соединения и связанные данные из PubChem.

---

## 1. Коротко о том, что мы делаем

### Формулировка задачи

Пользователь пишет естественным языком что-то вроде:

- "антибиотик с бензольным кольцом, молекулярная масса около 350"
- "соединение похоже на aspirin"
- "найди молекулу по описанию и верни свойства"

Система должна:

1. понять, что именно хочет пользователь;
2. преобразовать текстовый запрос в более конкретный машинный запрос;
3. вызвать нужные инструменты для PubChem;
4. вернуть пользователю подходящие соединения и основные свойства.

### Что является успешным результатом

На первой версии агент должен уметь:

- принять текстовый запрос;
- выделить из него полезные признаки;
- вызвать PubChem через tool calling;
- вернуть список кандидатов и базовые поля;
- объяснить, по каким признакам был найден результат.

---

## 2. Главный вывод со встречи

### Главное решение

**Первая версия должна быть без RAG.**

Фокус встречи был на том, что вам сейчас нужна **не “большая агентная система со всем сразу”**, а **рабочий tool-based MVP**:

- LLM + function calling / ReAct pattern;
- несколько узких тулов;
- маршрутизация между ними;
- запросы в PubChem;
- аккуратный ответ пользователю.

### Почему именно так

Потому что:

- это быстрее реализовать;
- это легче тестировать;
- это понятнее защищать;
- это не раздувает scope;
- это позволяет раньше получить working demo.

RAG обсуждался как идея для **второго этапа**, а не как обязательная часть первой версии.

---

## 3. Scope MVP

## Что входит в MVP

- Обертка над PubChem API.
- Перевод текущей обертки в набор тулов.
- Один агент с tool calling.
- Поддержка текстового запроса пользователя.
- Базовая интерпретация запроса через LLM.
- Возврат найденных соединений и основных свойств.
- Поддержка хотя бы английского языка на уровне API-вызовов.
- При необходимости: LLM-перевод русскоязычного запроса во внутреннее англоязычное представление.

## Что не входит в MVP

- Полноценный RAG.
- Векторная база.
- Иерархический RAG.
- Масштабная knowledge base поверх PubChem.
- Сложная мультиагентная архитектура.
- MCP-сервер как обязательная часть.
- Большое количество тулов.
- Попытка решить “все химические вопросы” в одном агенте.

### Практический критерий готовности MVP

Система проходит набор из 10-20 тестовых пользовательских запросов и:

- не падает;
- корректно вызывает тул;
- отдает разумный список кандидатов;
- показывает хотя бы базовую интерпретацию запроса;
- соблюдает rate limit;
- может поддерживать короткий диалог уточнения.

---

## 4. Архитектурная идея из встречи

## Рекомендуемая базовая схема

1. Пользователь пишет запрос естественным языком.
2. LLM извлекает из текста признаки:
   - название;
   - синоним;
   - молекулярная масса;
   - формула;
   - класс вещества;
   - возможные структурные признаки;
   - возможный SMILES, если запрос позволяет.
3. LLM выбирает подходящий тул.
4. Тул делает запрос в PubChem.
5. Результаты нормализуются.
6. LLM формирует человекочитаемый ответ.
7. Если данных недостаточно, агент задает уточняющий вопрос.

## Почему tools, а не жесткая логика

На встрече была ключевая мысль:

- даем LLM описание тулов;
- модель сама выбирает, какой тул вызвать;
- сама делает маршрутизацию;
- это и есть ReAct/tool-calling подход.

То есть вы не прошиваете все сценарии вручную, а даете модели набор безопасных, понятных, узких инструментов.
