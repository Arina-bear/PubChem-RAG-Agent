## 12. Подробный список задач

## Задача 1. Зафиксировать scope первой версии

### Что сделать

Сформулировать и заморозить MVP:

- без RAG;
- с tool calling;
- с 5-8 узкими тулами;
- с базовым диалогом;
- с поддержкой русскоязычного пользовательского ввода через LLM-нормализацию.

### Зачем

Чтобы не расползся scope и вы не ушли в инфраструктуру вместо рабочего демо.

### Подсказка

Запишите отдельно:

- `MVP`
- `post-MVP`
- `research backlog`

И не смешивайте это.

---

## Задача 2. Перевести текущую PubChem-обертку в тулы

### Что сделать

Разбить текущий код на отдельные функции-тулы.

### Минимально

- `search_compound_by_name`
- `search_compound_by_smiles`
- `search_compound_by_formula`
- `search_compound_by_mass_range`
- `get_compound_summary`
- `name_to_smiles`

### Подсказка

Каждый тул должен:

- делать одну вещь;
- иметь понятную сигнатуру;
- иметь хороший docstring;
- возвращать чистый JSON-friendly результат.

---

## Задача 3. Сделать агент поверх этих тулов

### Что сделать

Подключить LLM с function calling / tool calling и проверить, что она:

- понимает запрос;
- выбирает правильный тул;
- умеет вызвать 1-2 тула подряд;
- возвращает внятный ответ.

### Подсказка

Не стройте сложный граф сразу.  
Сначала обычный минимальный agent loop.

---

## Задача 4. Добавить слой интерпретации естественного языка

### Что сделать

Сделать структурированный парсер пользовательского запроса.

### На выходе парсера должно быть что-то вроде

```json
{
  "compound_name": null,
  "synonyms": ["antibiotic"],
  "mass_range": [330, 370],
  "contains_ring": "benzene",
  "language": "ru",
  "normalized_language": "en",
  "confidence": 0.62
}
```

### Подсказка

Не дайте LLM сразу идти в PubChem “как попало”.  
Сначала получите структурированную интерпретацию.

---

## Задача 5. Добавить механизм уточняющих вопросов

### Что сделать

Если запрос слишком абстрактный или неоднозначный, агент должен спрашивать уточнение.

### Примеры

- “Нужен поиск по названию вещества или по свойствам?”
- “Вы знаете примерную формулу или массу?”
- “Нужен один кандидат или список похожих соединений?”

### Подсказка

Это резко повысит качество и снизит галлюцинации.

---

## Задача 6. Сделать rate limiting и защиту от ошибок API

### Что сделать

- rate limiter;
- retry/backoff;
- обработку таймаутов;
- обработку пустых ответов;
- graceful fallback в ответе пользователю.

### Подсказка

Отдельно тестируйте:

- burst запросов;
- пустые результаты;
- некорректные параметры;
- превышение лимита.

---

## Задача 7. Продумать хранение диалога

### Что сделать

Хранить:

- историю сообщений;
- структурированную интерпретацию;
- вызовы тулов;
- найденные кандидаты;
- активный контекст текущего диалога.

### Подсказка

Для MVP достаточно SQLite или JSON persistence.

---

## Задача 8. Поддержать русский язык через внутреннюю англоязычную нормализацию

### Что сделать

Добавить шаг:

- русскоязычный запрос;
- перевод/нормализация в английское поисковое представление;
- вызов PubChem;
- ответ пользователю снова на русском.

### Подсказка

Сохраняйте и русский оригинал, и английскую нормализацию.  
Это очень поможет в дебаге.

---

## Задача 9. Собрать evaluation set

### Что сделать

Создать 10-20 тестовых запросов разных типов:

- явное название;
- синоним;
- масса;
- формула;
- частичное текстовое описание;
- неоднозначный запрос;
- запрос с русским текстом.

### Подсказка

Без evaluation вы не поймете, стало лучше или хуже.

---

## Задача 10. Решить, когда и зачем добавлять RAG

### Что сделать

Четко зафиксировать:

- какие вопросы текущие tools покрывают;
- какие вопросы не покрывают;
- какие именно поля PubChem требуют knowledge retrieval.

### Подсказка

Добавляйте RAG только если:

- у вас уже есть рабочий tool-based поиск;
- есть реальные вопросы, которые tools не решают;
- есть корпус и время на оценку качества.
