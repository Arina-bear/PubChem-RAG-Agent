## 5. Разбиение на тулы

## Основной принцип

**Тулы должны быть узкими, простыми и хорошо описанными.**

Чем уже тул, тем:

- ниже шанс неправильного вызова;
- легче тестирование;
- проще дебаг;
- понятнее поведение агента.

## Рекомендуемый минимальный набор тулов

### Tool 1. `search_compound_by_name`

Поиск соединения по названию или ключевому слову.

**Зачем нужен:**  
самый частый и понятный сценарий.

**Подсказка:**  
используйте его первым, если пользователь явно назвал вещество.

### Tool 2. `search_compound_by_smiles`

Поиск по SMILES.

**Зачем нужен:**  
это самый точный структурный путь, если удалось привести запрос к SMILES.

**Подсказка:**  
важный промежуточный инструмент, особенно если вы добавите LLM-преобразование “название -> SMILES”.

### Tool 3. `search_compound_by_formula`

Поиск по молекулярной формуле.

**Зачем нужен:**  
когда пользователь знает формулу или часть формулы.

### Tool 4. `search_compound_by_mass_range`

Поиск по диапазону массы.

**Зачем нужен:**  
в примерах из задания масса — один из ключевых признаков.

**Подсказка:**  
лучше сделать диапазон, а не точное равенство.

### Tool 5. `get_compound_summary`

Получение основных данных по найденному CID:

- canonical SMILES;
- molecular weight;
- molecular formula;
- IUPAC name;
- synonyms;
- краткие свойства.

**Зачем нужен:**  
разделить “поиск кандидатов” и “дозапрос деталей”.

### Tool 6. `name_to_smiles`

Конвертация названия вещества в SMILES через доступный источник/поиск.

**Зачем нужен:**  
это буквально было одной из ключевых идей со встречи:  
**сделать тул, который конвертирует название в SMILES**.

### Tool 7. `search_by_synonym`

Поиск по синонимам и альтернативным именам.

**Зачем нужен:**  
пользователи редко используют канонические названия.

### Tool 8. `ask_user_for_clarification`

Не “химический” тул, но полезный агентный механизм: если данных недостаточно, агент задает уточнение.

**Примеры:**

- “Вы знаете примерную молекулярную массу?”
- “Нужно искать по названию, структуре или фармакологическому описанию?”
- “Нужен один результат или список кандидатов?”

## 6. Как превратить обертку в tool calling

На встрече отдельно обсуждали, что текущую обертку нужно **перевести в тулы**.

## Практический принцип

Если у вас уже есть Python-функция, которая делает запрос к PubChem, то ее нужно:

1. сделать отдельной функцией;
2. описать понятным `docstring`;
3. дать минимальный и явный набор аргументов;
4. подключить к LLM как tool.

## Важная подсказка из встречи

`docstring` важен, потому что именно описание функции помогает модели понять:

- что делает тул;
- когда его вызывать;
- какие аргументы передавать.

То есть хороший docstring — это не “красота”, а часть интерфейса между LLM и вашим кодом.

## 7. ReAct pattern: как понимать его для вашего проекта

## Суть паттерна

Вы даете модели:

- системную роль;
- сообщение пользователя;
- список доступных тулов.

Модель:

1. анализирует задачу;
2. решает, какой тул вызвать;
3. получает результат;
4. при необходимости вызывает следующий тул;
5. формирует финальный ответ.

## Что это дает

- меньше ручной маршрутизации;
- легче поддерживать;
- удобнее расширять;
- можно постепенно добавлять новые тулы.

## Ограничение со встречи

Упоминалось, что современные системы комфортно работают **примерно до 30 тулов**.  
Поэтому вам сейчас не нужно проектировать огромную “супер-платформу”.

Для вашего кейса это значит:

- лучше 5-8 хороших тулов, чем 30 сырых;
- сначала делаем базовый набор;
- дальше, если нужно, добавляем оркестрацию.
